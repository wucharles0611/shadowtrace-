## 审查原则

1. 先理解需求目标、系统约定、调用链和数据流，再判断是否真满足需求。
2. 优先找真实可触发的问题；不为显得严格而编造。
3. 每个问题必须写清：触发条件、影响、代码为何会走到该路径、文件与行号、修复建议、建议测试。
4. 高风险面：
   - 边界：空输入、缺失依赖、权限失败、网络失败、并发、重复执行、超大输入
   - 异常：整批中断、吞错、误报成功、丢失进度
   - 一致性：缓存、checkpoint、事务、outbox、CAS、重试、临时文件
   - 兼容：旧 API / 配置 / 数据格式 / CLI
   - 安全：路径穿越、注入、权限绕过、敏感信息入日志/EventBus、不可信输入
   - 测试：是否覆盖真实失败路径，而非只 happy path
   - 工程：跨模块契约变更、是否需文档/ack
5. 主动构造失败场景（依赖抛错、中途失败再跑、并发乱序、配置缺字段、Provider 已 accepted 后崩溃等）。
6. 非 blocker 标为 **Should-Fix** 或 **Nit**，不要混级别。
